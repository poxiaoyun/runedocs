# 术语表

本页列出 **Rune Console** 平台中常用的核心术语及其含义，按功能领域分类整理。术语表旨在帮助用户、开发者及管理员快速理解平台各模块的关键概念。

> [!TIP]
> 使用浏览器的 `Ctrl+F`（macOS 下为 `Cmd+F`）可以快速检索本页术语。

## 平台概念 {#平台概念}

| 术语 | 英文 | 说明 |
|------|------|------|
| **晓石 AI** | XiaoShi AI | 晓石 AI 是整个 AI 平台的品牌名称，涵盖了从资源管理、模型开发、模型托管到在线推理的完整 AI 工程化能力。晓石 AI 由多个子产品组成，包括 Rune（AI 工作台）、Moha（模型中心）、ChatApp（对话体验）以及 LLM Gateway（大模型网关）等，为企业和开发者提供一站式 AI 研发与运营平台。 |
| **Rune Console** | Rune Console | Rune Console 是晓石 AI 平台的前端门户系统（Web UI），分为面向开发者的 **Console（租户门户）** 和面向平台管理员的 **BOSS（管理门户）** 两大界面。用户通过 Rune Console 完成模型训练、推理部署、资源管理、权限配置等所有操作。参见 [Console（租户门户）](#console) 和 [BOSS（管理门户）](#boss)。 |
| <span id="console">**Console（租户门户）**</span> | Console | Console 是面向 **租户管理员** 和 **开发者** 的日常操作界面。在 Console 中，用户可以管理工作空间、部署推理服务、提交微调任务、使用开发环境、浏览模型仓库以及调试 AI 对话等。Console 按照租户维度进行资源隔离，登录后用户需先选择所属租户再进入操作。参见 [租户](#租户)。 |
| <span id="boss">**BOSS（管理门户）**</span> | BOSS (Back-Office Support System) | BOSS 是面向 **系统管理员（平台管理员）** 的后台管理界面。通过 BOSS，管理员可以管理所有租户、集群、资源池、规格模板、系统应用、LLM 网关配置以及全局平台设置。BOSS 操作影响全局，通常只有少数高权限管理员可以访问。参见 [系统管理员](#系统管理员)。 |
| **Rune** | Rune | Rune 是 AI 工作台子产品，提供推理服务、微调训练、开发环境（IM）、应用部署、实验管理、模型评测等 AI 工作负载管理能力。Rune 建立在 Kubernetes 之上，支持多集群、多资源池的弹性调度。参见 [AI 工作负载](#ai-工作负载)。 |
| **Moha** | Moha (Model Hub & Assets) | Moha 是模型中心子产品，提供类似 Hugging Face 的模型仓库、数据集仓库、容器镜像管理以及在线 Space 展示功能。Moha 支持 Git LFS 大文件管理、分支与标签版本控制、Pull Request 协作流程以及从外部源（如 HuggingFace、ModelScope）镜像同步。参见 [模型中心](#模型中心)。 |
| **ChatApp（对话体验）** | ChatApp | ChatApp 是 AI 对话体验子产品，面向开发者和最终用户。它提供 AI 对话调试、多模型对比、沉浸式体验以及 Token 用量统计等功能。开发者可以通过 ChatApp 快速验证已部署的推理服务或 LLM 网关渠道的效果。参见 [渠道](#渠道)。 |
| **LLM 网关** | LLM Gateway | LLM 网关是大语言模型的统一调用代理层。它将多个上游 LLM 供应商（如 OpenAI、Azure、本地推理服务等）封装为统一的 API 接口，并提供 API Key 管理、渠道路由与降级、内容审核、限速策略、审计日志等企业级能力。参见 [LLM 网关](#llm-网关)。 |

## 资源层级 {#资源层级}

平台的资源管理采用层级化结构，从上至下依次为：**平台 → 租户 → 区域/集群 → 工作空间 → 实例**。

| 术语 | 英文 | 说明 |
|------|------|------|
| **平台** | Platform | 平台是整个晓石 AI 系统的最顶层概念，包含所有租户、集群和全局配置。平台级别的操作通过 BOSS 管理门户完成，例如创建租户、注册集群、管理全局规格模板等。一个平台部署可服务多个租户。参见 [BOSS（管理门户）](#boss)。 |
| <span id="租户">**租户**</span> | Tenant | 租户是业务隔离的顶层逻辑单元，通常对应一个组织、团队或业务部门。每个租户拥有独立的成员管理、工作空间列表、LLM 网关配置和资源配额。一个用户可以被分配到多个租户；登录时需要选择当前操作的租户。参见 [工作空间](#工作空间) 和 [配额](#配额)。 |
| **区域** | Region | 区域是集群的逻辑分组，通常按照地理位置（如"华东"、"北美"）或用途（如"生产"、"测试"）划分。区域帮助用户就近选择计算资源，降低网络延迟。一个区域下可包含多个集群。参见 [集群](#集群)。 |
| <span id="集群">**集群**</span> | Cluster | 集群对应一个 Kubernetes（K8s）集群，是实际承载计算资源的物理/虚拟基础设施。集群由平台管理员在 BOSS 中注册和维护，包含多个节点（Node）和资源池（Resource Pool）。集群支持 GPU、NPU 等异构算力。参见 [资源池](#资源池) 和 [节点](#节点)。 |
| <span id="工作空间">**工作空间**</span> | Workspace | 工作空间是平台中最小的资源隔离单元，也是开发者日常工作的核心容器。在一个工作空间中，用户可以创建推理服务、微调任务、开发环境、应用等各类 AI 工作负载实例。工作空间位于某个集群之上，并受资源配额的约束。参见 [实例](#实例) 和 [配额](#配额)。 |
| <span id="实例">**实例**</span> | Instance | 实例是所有 AI 工作负载的统称，包括推理服务、微调任务、开发环境、应用、实验、评测等。每个实例运行在特定工作空间内，消耗该工作空间的资源配额。实例的生命周期包括创建、运行、停止、删除等状态。参见 [AI 工作负载](#ai-工作负载)。 |
| <span id="命名空间">**命名空间**</span> | Namespace | 命名空间是 Kubernetes 中的资源隔离机制。在晓石 AI 平台中，每个工作空间通常映射到一个 K8s Namespace，实现不同工作空间之间的网络和资源隔离。用户一般无需直接操作 Namespace，由平台自动管理。 |

## AI 工作负载 {#ai-工作负载}

AI 工作负载是用户在工作空间中创建的各类计算任务和服务的总称。

| 术语 | 英文 | 说明 |
|------|------|------|
| <span id="推理服务">**推理服务**</span> | Inference Service | 推理服务用于将已训练好的 AI 模型部署为在线服务（HTTP/gRPC 端点），对外提供实时预测能力。用户可以指定模型来源（Moha 仓库或存储卷中的模型文件）、推理框架（如 vLLM、TGI、Triton 等）以及计算规格。推理服务支持弹性伸缩、版本管理和流量分发。部署后的推理服务还可以注册到 LLM 网关，统一对外暴露。参见 [规格](#规格) 和 [LLM 网关](#llm-网关)。 |
| **微调** | Fine-tuning | 微调是在预训练模型的基础上，使用特定领域数据进行进一步训练的过程。平台支持创建微调任务，用户可选择基础模型、训练数据集、训练参数（如学习率、epoch 数）和计算规格。微调任务完成后，可将产出的模型文件保存到存储卷或推送至 Moha 模型仓库。参见 [数据集](#数据集) 和 [存储卷](#存储卷)。 |
| **开发环境（IM）** | Dev Environment / Interactive Machine (IM) | 开发环境为用户提供交互式的在线编程空间，支持 JupyterLab、VS Code Server 等多种 IDE。用户可以在开发环境中编写代码、调试模型、浏览数据，并直接挂载存储卷来访问模型文件和数据集。开发环境适合探索性研究、原型开发和数据分析等场景。参见 [存储卷](#存储卷)。 |
| **应用** | Application | 应用是通过产品模板（Helm Chart）部署到工作空间中的容器化应用实例。用户可以从应用市场选择模板，填写配置参数后一键部署。应用可以是 Web 服务、API 后端、数据处理管道或任何容器化的工作负载。参见 [应用市场](#应用市场) 和 [产品模板](#产品模板)。 |
| <span id="实验">**实验**</span> | Experiment | 实验（Experiment）是机器学习实验管理功能，用于系统地跟踪和比较不同训练运行的参数配置、评估指标和输出结果。用户可以记录超参数、损失曲线、评测分数等信息，便于后续回溯和复现最优模型。实验管理帮助团队在大量训练迭代中保持有序。参见 [评测](#评测)。 |
| <span id="评测">**评测**</span> | Evaluation | 评测是对 AI 模型进行系统化质量评估的过程。平台提供标准化的评测流程，支持自定义评测指标、数据集和评分规则。评测结果可以帮助开发者量化模型性能，比较不同模型版本的优劣，从而指导模型优化和上线决策。参见 [实验](#实验)。 |
| <span id="存储卷">**存储卷**</span> | Storage Volume | 存储卷是平台提供的持久化存储资源。用户可以创建存储卷并将其挂载到推理服务、微调任务、开发环境等实例中，用于存放模型文件、训练数据、代码和输出结果。存储卷的数据在实例重启或重建后依然保留，确保数据持久性。参见 [存储任务](#存储任务) 和 [存储集群](#存储集群)。 |
| <span id="存储任务">**存储任务**</span> | Storage Job | 存储任务是向存储卷中导入数据的自动化任务。平台支持多种存储任务类型，包括从 Git 仓库克隆、从 HuggingFace/ModelScope 下载模型或数据集、从 Moha 仓库同步、创建 Python 虚拟环境等。存储任务在后台异步执行，完成后数据即可在存储卷中使用。参见 [存储卷](#存储卷)。 |

## 模型中心（Moha） {#模型中心}

Moha 是平台内置的模型与资产管理中心，提供类似 Hugging Face Hub 的功能。

| 术语 | 英文 | 说明 |
|------|------|------|
| **模型仓库** | Model Repository | 模型仓库是存放机器学习模型文件的版本化存储库。每个模型仓库基于 Git 进行版本控制，支持分支（Branch）、标签（Tag）和提交（Commit）管理。模型文件（如权重文件 `.bin`、`.safetensors`）通过 Git LFS 管理，支持大文件的高效上传和下载。参见 [Git LFS](#git-lfs) 和 [分支](#分支)。 |
| <span id="数据集">**数据集**</span> | Dataset | 数据集仓库用于托管 AI 训练和评测所需的数据文件。与模型仓库类似，数据集仓库也基于 Git 进行版本管理，支持分支、标签和 LFS 大文件存储。用户可在微调任务或评测任务中直接引用 Moha 中的数据集。参见 [微调](#微调) 和 [评测](#评测)。 |
| **镜像** | Image | 镜像是容器镜像仓库，用于托管 Docker/OCI 格式的容器镜像。用户可以推送自定义的运行环境镜像（例如包含特定框架和依赖的训练镜像），并在创建推理服务、微调任务或开发环境时选用。平台也提供预置的官方镜像。 |
| **展示空间（Space）** | Space | Space 是 Moha 提供的在线展示功能，允许用户为模型创建交互式的演示界面。类似 Hugging Face Spaces，用户可以上传 Gradio、Streamlit 等应用代码，平台自动构建并部署为可公开访问的 Web 应用，方便展示模型效果、收集用户反馈。 |
| **组织** | Organization | 组织是 Moha 中的协作单元，类似 GitHub 的 Organization。用户可以创建组织并邀请成员，以组织名义发布模型和数据集。组织下的仓库可设置不同的可见性和访问权限，便于团队协作管理 AI 资产。 |
| **讨论** | Discussion | 讨论是仓库（模型/数据集）中的社区互动功能。用户可以在仓库页面发起讨论帖，就模型用法、数据质量、Bug 反馈等话题进行交流。讨论支持 Markdown 格式和评论回复，促进社区协作。 |
| **合并请求（PR）** | Pull Request | 合并请求是 Git 式的协作流程，允许用户在分支上修改文件后，向主分支提交合并申请。仓库维护者可以审查变更内容、讨论修改并最终合并或拒绝请求。这一机制确保了对仓库内容的受控变更和质量保障。参见 [分支](#分支)。 |
| <span id="镜像同步">**镜像同步**</span> | Mirror | 镜像同步是 Moha 的数据同步功能，允许管理员将外部平台（如 Hugging Face、ModelScope）上的模型或数据集自动同步到本地 Moha 仓库。同步可设置为定时执行或手动触发，确保本地仓库与上游保持一致。这在网络受限环境下尤为实用。 |
| <span id="分支">**分支**</span> | Branch | 分支是 Git 版本控制中的核心概念。在 Moha 仓库中，分支允许用户在不影响主分支（main）的情况下进行修改和实验。开发完成后，可通过合并请求将分支变更合入主分支。 |
| **提交** | Commit | 提交是 Git 仓库中的一次原子性变更记录，包含修改的文件、作者信息和提交消息。每次提交都有唯一的哈希值标识，用户可以回溯到任意历史提交查看仓库在该时间点的状态。 |
| **标签** | Tag | 标签是对某个特定提交打上的有意义的名称标记，通常用于标注版本号（如 `v1.0`、`v2.1-beta`）。标签使用户可以快速定位和下载特定版本的模型或数据集。 |
| <span id="git-lfs">**Git LFS**</span> | Git Large File Storage | Git LFS 是 Git 大文件存储扩展。由于 AI 模型权重文件通常体积巨大（数 GB 甚至数百 GB），直接存入 Git 仓库会导致性能问题。Git LFS 通过将大文件存储在专用的对象存储中，在 Git 仓库中仅保留轻量级指针文件，从而实现高效的大文件版本管理。 |

## LLM 网关 {#llm-网关}

LLM 网关为大语言模型调用提供统一的代理层和企业级治理能力。

| 术语 | 英文 | 说明 |
|------|------|------|
| <span id="渠道">**渠道**</span> | Channel | 渠道代表一个 LLM 服务提供方的接入配置，定义了上游 API 地址、认证信息、支持的模型列表和路由权重等参数。一个网关可以配置多个渠道（如 OpenAI、Azure OpenAI、本地 vLLM 推理服务等），实现多供应商统一接入。用户请求到达网关后，根据请求的模型名称和路由规则分发到对应的渠道。参见 [渠道降级](#渠道降级) 和 [供应商](#供应商)。 |
| <span id="供应商">**供应商**</span> | Provider | 供应商是渠道背后实际提供 LLM 推理能力的服务商或系统，如 OpenAI、Anthropic、Azure、Google、阿里通义、百度文心或平台内部的推理服务。每个供应商有其特定的 API 协议和鉴权方式，网关通过渠道配置进行协议适配。 |
| **API Key / Token** | API Key / Token | API Key（也叫 Token）是调用 LLM 网关 API 的身份凭证。管理员或开发者可以创建 API Key，并为其设置名称、过期时间和可调用的模型范围。客户端在请求头中携带 API Key 即可调用网关代理的 LLM 服务。每个 API Key 的调用记录均会被审计。参见 [审计日志](#审计日志)。 |
| <span id="审计日志">**审计日志**</span> | Audit Log | 审计日志记录通过 LLM 网关发生的每一次 API 调用详情，包括请求时间、调用者（API Key）、请求的模型、输入/输出 Token 数量、响应状态码、耗时以及是否触发内容审核等信息。审计日志帮助管理员监控使用情况、排查问题和进行合规审计。参见 [用量记录](#用量记录)。 |
| **审查策略** | Moderation Policy | 审查策略定义了 LLM 网关对请求和响应内容进行安全检查的规则。管理员可以创建不同粒度的策略，指定检查方式（如关键词匹配、正则表达式、AI 审核模型）、触发条件和处理动作（拦截、警告、记录等）。审查策略可按渠道或全局维度绑定。参见 [内容审核](#内容审核) 和 [词库](#词库)。 |
| <span id="词库">**词库**</span> | Lexicon | 词库是内容审核中使用的敏感词/关键词集合。管理员可以维护多个词库，将不同类别的敏感词分组管理（如涉政词汇、色情内容、广告信息等）。词库在审查策略中被引用，用于对请求或响应文本进行关键词匹配检查。 |
| <span id="内容审核">**内容审核**</span> | Content Moderation | 内容审核是 LLM 网关的安全防护能力，在请求进入上游模型之前或响应返回用户之前，检查文本内容是否包含违规信息。审核可基于词库匹配、正则规则或外部审核 AI 模型进行。命中规则后可自动拦截请求或脱敏处理。参见 [审查策略](#审查策略)。 |
| **限速（RPM/TPM）** | Rate Limiting (RPM/TPM) | 限速是对 LLM 网关 API 调用频率的控制机制。RPM（Requests Per Minute）限制每分钟的请求数量，TPM（Tokens Per Minute）限制每分钟消耗的 Token 总量。限速策略可按 API Key、渠道或模型维度配置，防止资源被过度消耗或滥用。 |
| <span id="渠道降级">**渠道降级**</span> | Channel Fallback | 渠道降级是 LLM 网关的高可用机制。当某个渠道出现故障（如上游 API 不可用、超时或报错率过高）时，网关自动将请求路由到备选渠道，保障服务不中断。管理员可以配置降级顺序和触发条件。参见 [渠道](#渠道)。 |
| <span id="用量记录">**用量记录**</span> | Usage Record | 用量记录汇总统计 LLM 网关的资源消耗情况，包括各 API Key、各模型、各渠道的调用次数和 Token 使用量。用量数据可用于成本核算、容量规划和优化建议。参见 [审计日志](#审计日志)。 |
| **服务注册** | Service Registration | 服务注册是将平台内部署的推理服务接入 LLM 网关的过程。注册后，推理服务作为一个渠道出现在网关中，可统一通过网关 API Key 进行调用、审计和管控，无需直接暴露推理服务端点。参见 [推理服务](#推理服务) 和 [渠道](#渠道)。 |

## 用户与权限 {#用户与权限}

平台采用基于角色的多层级权限体系（RBAC），覆盖平台、租户和工作空间三个层级。

| 术语 | 英文 | 说明 |
|------|------|------|
| <span id="系统管理员">**系统管理员**</span> | System Admin | 系统管理员是平台的最高权限角色，可通过 BOSS 管理门户执行所有操作，包括创建/删除租户、注册/管理集群、配置全局规格和系统模板、管理 LLM 网关全局配置等。系统管理员通常是平台运维团队的核心成员。参见 [BOSS（管理门户）](#boss)。 |
| **租户管理员** | Tenant Admin | 租户管理员负责管理其所在租户的全部资源和成员。权限包括创建/删除工作空间、邀请/移除成员、分配角色、管理租户级 LLM 网关配置、查看租户资源用量等。一个租户可以有多个租户管理员。参见 [租户](#租户)。 |
| **开发者** | Developer | 开发者是工作空间中的核心使用角色，可以在授权的工作空间中创建和管理各类 AI 工作负载实例（推理服务、微调、开发环境、应用等）。开发者可以读写存储卷、提交实验、执行评测，但通常无法管理工作空间的成员或配额配置。 |
| **成员** | Member | 成员是最基础的权限角色，拥有对工作空间资源的只读访问权限。成员可以查看实例状态、浏览日志、查看存储卷内容等，但不能创建或修改资源。成员角色适合用于需要了解项目进展但不参与操作的人员。 |
| **RBAC** | Role-Based Access Control | 基于角色的访问控制是平台权限管理的核心机制。平台预定义了系统管理员、租户管理员、开发者、成员等角色，每个角色绑定了一组细粒度权限。权限在平台、租户、工作空间三个层级分别管理，确保最小权限原则的落实。参见 [权限表达式](#权限表达式)。 |
| <span id="权限表达式">**权限表达式**</span> | Permission Expression | 权限表达式（如 `rune:inference:create`、`moha:model:read`）是平台中细粒度权限的编码格式。每个表达式由"子产品:资源类型:操作"三段组成，用于精确控制用户在特定资源上的操作能力。角色由一组权限表达式组合而成。 |
| **MFA** | Multi-Factor Authentication | 多因素认证（MFA）是增强登录安全性的机制，在用户名/密码之外增加第二重验证因素（如 TOTP 动态验证码）。平台支持为用户账户启用 MFA，登录时需同时提供密码和动态验证码，有效防止账户被盗用。 |
| **API Key（用户级）** | API Key (User-Level) | 用户级 API Key 是用户用于程序化访问平台 REST API 的身份凭证，区别于 LLM 网关的 API Key。用户可以在个人设置中生成 API Key，设置有效期和权限范围，然后在 CLI 工具或自动化脚本中使用。参见 [API Key / Token](#api-key--token)。 |
| **SSH Key** | SSH Key | SSH Key 是基于公钥密码学的认证方式，用户将公钥上传至平台后，可通过 SSH 协议进行 Git 仓库的推送/拉取操作（无需每次输入密码）。SSH Key 在 Moha 模型仓库和数据集仓库的 Git 操作中尤为常用。 |

## 计算资源 {#计算资源}

平台的计算资源管理围绕集群、资源池、规格和配额四大核心概念组织。

| 术语 | 英文 | 说明 |
|------|------|------|
| <span id="规格">**规格（Flavor）**</span> | Flavor | 规格是预定义的计算资源配置模板，定义了一个实例可使用的 CPU 核数、内存大小和加速卡（GPU/NPU 等）数量的组合。例如"4C 16G 1×A100"表示 4 核 CPU、16 GB 内存、1 块 A100 GPU。平台管理员在 BOSS 中统一管理规格模板，租户和开发者在创建实例时从可用规格列表中选择。参见 [配额](#配额)。 |
| <span id="资源池">**资源池**</span> | Resource Pool | 资源池是集群中的一组节点（Node）的逻辑分组，用于实现资源隔离和差异化调度。例如，可将 GPU 节点和 CPU 节点分入不同资源池，或为不同租户划分专用资源池。工作空间在创建时绑定到特定资源池，其内所有实例只能调度到该资源池的节点上。参见 [节点](#节点)。 |
| <span id="配额">**配额**</span> | Quota | 配额是对资源使用量的上限控制，防止单个租户或工作空间过度占用共享资源。平台支持三级配额体系：**集群级配额**（限制租户在某集群中的总资源量）、**租户级配额**（限制租户的全局资源量）和**工作空间级配额**（限制某个工作空间的资源量）。配额通常以 CPU 核数、内存 GB 数和加速卡数量为计量单位。 |
| **GPU** | Graphics Processing Unit | GPU（图形处理单元）是 AI 训练和推理中最常用的加速硬件。平台支持 NVIDIA GPU（如 A100、H100、V100）等多种型号，通过资源池和规格进行管理和分配。 |
| **NPU** | Neural Processing Unit | NPU（神经网络处理单元）是专为 AI 推理和训练优化的专用芯片，如华为昇腾（Ascend）系列。平台对 NPU 的支持方式与 GPU 类似，通过规格和资源池进行管理。 |
| **DCU** | Deep Computing Unit | DCU（深度计算单元）是海光信息推出的通用 GPU 加速卡，在国产化场景中用于替代 NVIDIA GPU。平台支持 DCU 加速卡的调度和管理。 |
| **MLU** | Machine Learning Unit | MLU（机器学习单元）是寒武纪推出的 AI 加速芯片。平台将 MLU 作为一种与 GPU 并列的加速器类型进行资源管理和调度。 |
| **vGPU** | Virtual GPU | vGPU（虚拟 GPU）是将一块物理 GPU 虚拟化为多个逻辑 GPU 的技术，允许多个实例共享同一块物理 GPU 的算力和显存。vGPU 适合对 GPU 资源需求较小的场景（如轻量推理、开发调试），能显著提高 GPU 利用率。 |
| **Pod** | Pod | Pod 是 Kubernetes 中最小的可调度单元，包含一个或多个容器。平台中的每个实例（推理服务、微调任务等）在 K8s 集群中以 Pod 形式运行。用户通常无需直接操作 Pod，但可以在实例详情中查看 Pod 状态和日志。 |
| <span id="节点">**节点**</span> | Node | 节点是 Kubernetes 集群中的一台物理或虚拟服务器。节点提供 CPU、内存和加速卡等计算资源，Pod 被调度到节点上运行。节点按角色可分为管理节点（Master）和工作节点（Worker）。参见 [集群](#集群)。 |

## 应用模板与市场 {#应用模板与市场}

平台通过模板化机制实现应用的标准化部署和分发。

| 术语 | 英文 | 说明 |
|------|------|------|
| <span id="产品模板">**产品模板**</span> | Product (Template) | 产品模板是应用部署的蓝图，定义了应用的容器镜像、资源需求、配置参数和部署方式。每个模板基于 Helm Chart 构建，并附带版本管理。模板分为系统模板（所有租户可见）和管理员模板（特定租户可见）两类。参见 [Helm Chart](#helm-chart)。 |
| <span id="helm-chart">**Helm Chart**</span> | Helm Chart | Helm Chart 是 Kubernetes 应用打包标准，将应用的 YAML 资源清单、配置模板和默认参数封装为一个可复用的部署包。晓石 AI 的产品模板底层采用 Helm Chart 格式，支持版本发布和升级回滚。 |
| **SchemaForm（动态表单）** | SchemaForm | SchemaForm 是平台根据产品模板的配置参数定义（JSON Schema）自动生成的动态表单。用户在部署应用时，无需手动编辑 YAML 文件，而是通过可视化表单填写配置参数（如副本数、镜像版本、环境变量等），降低了部署门槛。 |
| **版本** | Version | 产品模板的版本号（如 `1.0.0`、`2.3.1`），标识模板的特定发布。每次更新模板内容后发布新版本，用户可以在部署时选择特定版本或升级到最新版本。版本管理确保了模板变更的可追溯性和稳定性。 |
| <span id="应用市场">**应用市场**</span> | App Market | 应用市场是面向开发者和工作空间管理员的模板浏览和部署入口。用户可以在应用市场中搜索和筛选可用的产品模板，查看模板说明和配置参数，然后一键部署到指定工作空间。应用市场实现了应用的自助化分发。参见 [产品模板](#产品模板)。 |
| **系统模板** | System Product | 系统模板是由平台管理员通过 BOSS 创建和管理的产品模板，对所有租户可见。系统模板通常包含平台推荐的、经过验证的标准应用（如常用推理框架、监控工具、ML 流水线组件等）。 |
| **管理员模板** | Admin Product | 管理员模板是由租户管理员创建和管理的产品模板，仅对该租户下的工作空间可见。管理员模板允许租户根据自身业务需求定制专属应用模板，如内部工具、私有服务等。 |

平台提供多层次的存储管理能力，支撑 AI 工作负载的数据持久化和文件管理需求。

| 术语 | 英文 | 说明 |
|------|------|------|
| **存储卷** | Storage Volume | 参见 [AI 工作负载 - 存储卷](#存储卷)。存储卷是实例的持久化存储介质，可在多个实例之间共享数据。 |
| <span id="存储集群">**存储集群**</span> | Storage Cluster | 存储集群是平台底层的分布式存储基础设施，为存储卷提供实际的存储空间。平台管理员在 BOSS 中注册和管理存储集群。一个计算集群可关联多个存储集群，以满足不同的容量和性能需求。 |
| **S3 文件代理** | S3 File Proxy | S3 文件代理是平台提供的文件访问代理服务，将底层存储（如 Ceph、MinIO 等）的文件通过兼容 S3 协议的接口暴露给用户或实例。用户可通过平台文件管理器或 S3 客户端工具直接上传、下载和管理存储卷中的文件。 |

| **存储任务类型** | Storage Job Kinds | 平台支持以下几种存储任务类型：<br/>• **Git**：从 Git 仓库克隆代码或文件到存储卷中。<br/>• **HuggingFace**：从 Hugging Face Hub 下载模型或数据集到存储卷中。<br/>• **ModelScope**：从魔搭社区（ModelScope）下载模型或数据集到存储卷中。<br/>• **PythonEnv**：在存储卷中创建 Python 虚拟环境并安装指定依赖。<br/>• **Moha**：从平台内置的 Moha 仓库同步模型或数据集到存储卷中。<br/>参见 [存储任务](#存储任务)。 |

## 可观测性 {#可观测性}

平台提供全方位的可观测性能力，帮助用户和管理员监控资源使用、排查运行问题。

| 术语 | 英文 | 说明 |
|------|------|------|
| **仪表盘** | Dashboard | 仪表盘是平台的概览信息页面，展示关键统计指标和资源摘要。Console 仪表盘展示租户级工作空间概况、实例统计和资源使用率；BOSS 仪表盘展示平台级的集群状态、租户数量、全局资源使用情况等。 |
| **动态仪表盘** | Dynamic Dashboard | 动态仪表盘是 BOSS 中基于 Grafana 或类似引擎的可定制监控面板。管理员可以配置自定义的指标查询和可视化图表，实时展示集群的 CPU/GPU 利用率、内存消耗、Pod 调度情况等关键运维指标。参见 [指标](#指标)。 |
| **日志查看器** | Log Viewer | 日志查看器是实例详情页中的日志浏览工具。用户可以查看实例中各容器的标准输出/错误日志（stdout/stderr），支持关键词搜索、时间范围筛选和日志下载。日志查看器帮助开发者快速排查实例运行中的错误和异常。参见 [日志流](#日志流)。 |
| <span id="日志流">**日志流**</span> | Log Stream (WebSocket) | 日志流通过 WebSocket 协议实现实例日志的实时推送。当用户打开日志查看器时，新产生的日志条目会实时追加到页面中，无需手动刷新。这对于监控正在运行的训练任务或服务启动过程尤为有用。 |
| <span id="指标">**指标**</span> | Metrics | 指标是量化系统运行状态的时序数据，如 CPU 利用率、GPU 显存使用比例、网络吞吐量、请求延迟等。平台从 Kubernetes 和各实例中采集指标数据，可在动态仪表盘和实例详情中展示。指标数据通常基于 Prometheus 体系采集和存储。 |
| **事件** | Events | 事件是 Kubernetes 中记录的系统事件信息，如 Pod 调度成功/失败、镜像拉取完成、容器重启等。用户可以在实例详情页查看相关事件，了解实例的创建过程和运行状态变化。事件信息对排查实例启动失败等问题非常有帮助。 |
| **监控** | Monitoring | 监控是对平台和实例运行状态进行持续观察的综合能力，涵盖指标采集、日志聚合、事件追踪和告警通知等方面。平台在多个层面提供监控支持：集群级监控（通过动态仪表盘）、实例级监控（通过日志查看器和指标图表）、网关级监控（通过审计日志和用量记录）。 |
