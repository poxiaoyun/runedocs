# Glossary

This page lists the core terms and their definitions commonly used in the **Rune Console** platform, organized by functional domain. The glossary is intended to help users, developers, and administrators quickly understand the key concepts of each platform module.

> [!TIP]
> Use your browser's `Ctrl+F` (or `Cmd+F` on macOS) to quickly search for terms on this page.

## Resource Hierarchy {#resource-hierarchy}

The platform's resource management uses a hierarchical structure, from top to bottom: **Platform → Tenant → Region/Cluster → Workspace → Instance**.

| Term | English | Description |
|------|---------|-------------|
| **Platform** | Platform | The platform is the topmost concept in the entire XiaoShi AI system, encompassing all tenants, clusters, and global configurations. Platform-level operations are performed through the BOSS admin portal, such as creating tenants, registering clusters, and managing global flavor templates. A single platform deployment can serve multiple tenants. See [BOSS (Admin Portal)](#boss). |
| <span id="tenant">**Tenant**</span> | Tenant | A tenant is the top-level logical unit for business isolation, typically corresponding to an organization, team, or business department. Each tenant has independent member management, workspace lists, LLM gateway configurations, and resource quotas. A user can be assigned to multiple tenants; they must select the current operating tenant at login. See [Workspace](#workspace) and [Quota](#quota). |
| **Region** | Region | A region is a logical grouping of clusters, typically organized by geographic location (e.g., "East China", "North America") or purpose (e.g., "Production", "Testing"). Regions help users select nearby compute resources to reduce network latency. A region can contain multiple clusters. See [Cluster](#cluster). |
| <span id="cluster">**Cluster**</span> | Cluster | A cluster corresponds to a Kubernetes (K8s) cluster, which is the physical/virtual infrastructure that actually hosts compute resources. Clusters are registered and maintained by platform administrators in BOSS, containing multiple nodes and resource pools. Clusters support heterogeneous computing including GPUs and NPUs. See [Resource Pool](#resource-pool) and [Node](#node). |
| <span id="workspace">**Workspace**</span> | Workspace | A workspace is the smallest resource isolation unit in the platform and the core container for developers' daily work. Within a workspace, users can create various AI workload instances such as inference services, fine-tuning tasks, development environments, and applications. Workspaces reside on a specific cluster and are constrained by resource quotas. See [Instance](#instance) and [Quota](#quota). |
| <span id="instance">**Instance**</span> | Instance | Instance is the collective term for all AI workloads, including inference services, fine-tuning tasks, development environments, applications, experiments, and evaluations. Each instance runs within a specific workspace and consumes that workspace's resource quota. An instance's lifecycle includes states such as created, running, stopped, and deleted. See [AI Workloads](#ai-workloads). |
| <span id="namespace">**Namespace**</span> | Namespace | A namespace is the resource isolation mechanism in Kubernetes. In the XiaoShi AI platform, each workspace typically maps to a K8s Namespace, achieving network and resource isolation between different workspaces. Users generally do not need to operate namespaces directly, as they are managed automatically by the platform. |

## AI Workloads {#ai-workloads}

AI workloads are the collective term for various compute tasks and services that users create within workspaces.

| Term | English | Description |
|------|---------|-------------|
| <span id="inference-service">**Inference Service**</span> | Inference Service | An inference service is used to deploy trained AI models as online services (HTTP/gRPC endpoints) that provide real-time prediction capabilities. Users can specify model sources (model files from Moha repositories or storage volumes), inference frameworks (such as vLLM, TGI, Triton), and compute flavors. Inference services support elastic scaling, version management, and traffic distribution. Deployed inference services can also be registered to the LLM Gateway for unified external exposure. See [Flavor](#flavor) and [LLM Gateway](#llm-gateway). |
| **Fine-tuning** | Fine-tuning | Fine-tuning is the process of further training a pre-trained model using domain-specific data. The platform supports creating fine-tuning tasks where users can select base models, training datasets, training parameters (such as learning rate, number of epochs), and compute flavors. After a fine-tuning task completes, the resulting model files can be saved to storage volumes or pushed to Moha model repositories. See [Dataset](#dataset) and [Storage Volume](#storage-volume). |
| **Dev Environment (IM)** | Dev Environment / Interactive Machine (IM) | Development environments provide users with interactive online programming spaces supporting multiple IDEs such as JupyterLab and VS Code Server. Users can write code, debug models, browse data, and directly mount storage volumes to access model files and datasets. Dev environments are suitable for exploratory research, prototype development, and data analysis. See [Storage Volume](#storage-volume). |
| **Application** | Application | An application is a containerized application instance deployed to a workspace through a product template (Helm Chart). Users can select templates from the App Market, fill in configuration parameters, and deploy with one click. Applications can be web services, API backends, data processing pipelines, or any containerized workload. See [App Market](#app-market) and [Product Template](#product-template). |
| <span id="experiment">**Experiment**</span> | Experiment | Experiments provide machine learning experiment management functionality for systematically tracking and comparing parameter configurations, evaluation metrics, and output results across different training runs. Users can record hyperparameters, loss curves, evaluation scores, and other information for retrospective analysis and reproducing optimal models. Experiment management helps teams stay organized across numerous training iterations. See [Evaluation](#evaluation). |
| <span id="evaluation">**Evaluation**</span> | Evaluation | Evaluation is the process of systematically assessing AI model quality. The platform provides standardized evaluation workflows supporting custom evaluation metrics, datasets, and scoring rules. Evaluation results help developers quantify model performance, compare different model versions, and guide model optimization and deployment decisions. See [Experiment](#experiment). |
| <span id="storage-volume">**Storage Volume**</span> | Storage Volume | A storage volume is a persistent storage resource provided by the platform. Users can create storage volumes and mount them to instances such as inference services, fine-tuning tasks, and development environments for storing model files, training data, code, and output results. Storage volume data persists across instance restarts or rebuilds, ensuring data durability. See [Storage Job](#storage-job) and [Storage Cluster](#storage-cluster). |
| <span id="storage-job">**Storage Job**</span> | Storage Job | A storage job is an automated task for importing data into storage volumes. The platform supports multiple storage job types including cloning from Git repositories, downloading models or datasets from HuggingFace/ModelScope, syncing from Moha repositories, and creating Python virtual environments. Storage jobs run asynchronously in the background, and data is available in the storage volume once completed. See [Storage Volume](#storage-volume). |

## Model Hub (Moha) {#model-hub}

Moha is the platform's built-in model and asset management center, providing functionality similar to Hugging Face Hub.

| Term | English | Description |
|------|---------|-------------|
| **Model Repository** | Model Repository | A model repository is a versioned storage for machine learning model files. Each model repository uses Git for version control, supporting branch, tag, and commit management. Model files (such as weight files `.bin`, `.safetensors`) are managed through Git LFS, enabling efficient upload and download of large files. See [Git LFS](#git-lfs) and [Branch](#branch). |
| <span id="dataset">**Dataset**</span> | Dataset | Dataset repositories are used to host data files needed for AI training and evaluation. Similar to model repositories, dataset repositories also use Git for version management, supporting branches, tags, and LFS large file storage. Users can directly reference datasets from Moha in fine-tuning or evaluation tasks. See [Fine-tuning](#fine-tuning) and [Evaluation](#evaluation). |
| **Image** | Image | Images refer to the container image registry for hosting Docker/OCI format container images. Users can push custom runtime environment images (e.g., training images with specific frameworks and dependencies) and select them when creating inference services, fine-tuning tasks, or development environments. The platform also provides pre-built official images. |
| **Space** | Space | Space is an online showcase feature provided by Moha, allowing users to create interactive demo interfaces for models. Similar to Hugging Face Spaces, users can upload Gradio, Streamlit, or other application code, and the platform automatically builds and deploys it as a publicly accessible web application for demonstrating model capabilities and collecting user feedback. |
| **Organization** | Organization | An organization is a collaborative unit in Moha, similar to a GitHub Organization. Users can create organizations and invite members to publish models and datasets under the organization's name. Repositories under an organization can have different visibility and access permissions, facilitating team collaboration in managing AI assets. |
| **Discussion** | Discussion | Discussions are community interaction features within repositories (models/datasets). Users can initiate discussion threads on repository pages to exchange ideas about model usage, data quality, bug reports, and more. Discussions support Markdown formatting and comment replies to promote community collaboration. |
| **Pull Request (PR)** | Pull Request | Pull Requests are Git-style collaboration workflows that allow users to modify files on a branch and submit a merge request to the main branch. Repository maintainers can review changes, discuss modifications, and ultimately merge or reject the request. This mechanism ensures controlled changes and quality assurance for repository content. See [Branch](#branch). |
| <span id="mirror-sync">**Mirror Sync**</span> | Mirror | Mirror sync is Moha's data synchronization feature that allows administrators to automatically sync models or datasets from external platforms (such as Hugging Face, ModelScope) to local Moha repositories. Syncing can be set to run on a schedule or triggered manually, ensuring local repositories stay consistent with upstream. This is particularly useful in network-restricted environments. |
| <span id="branch">**Branch**</span> | Branch | A branch is a core concept in Git version control. In Moha repositories, branches allow users to make modifications and experiments without affecting the main branch. After development is complete, branch changes can be merged into the main branch via Pull Requests. |
| **Commit** | Commit | A commit is an atomic change record in a Git repository, containing modified files, author information, and a commit message. Each commit has a unique hash identifier, allowing users to trace back to any historical commit and view the repository's state at that point in time. |
| **Tag** | Tag | A tag is a meaningful name marker applied to a specific commit, typically used to annotate version numbers (e.g., `v1.0`, `v2.1-beta`). Tags enable users to quickly locate and download specific versions of models or datasets. |
| <span id="git-lfs">**Git LFS**</span> | Git Large File Storage | Git LFS is a Git extension for large file storage. Since AI model weight files are typically very large (several GB to hundreds of GB), storing them directly in Git repositories would cause performance issues. Git LFS stores large files in dedicated object storage while keeping only lightweight pointer files in the Git repository, enabling efficient large file version management. |

## LLM Gateway {#llm-gateway}

The LLM Gateway provides a unified proxy layer and enterprise-grade governance capabilities for large language model invocations.

| Term | English | Description |
|------|---------|-------------|
| <span id="channel">**Channel**</span> | Channel | A channel represents the integration configuration for an LLM service provider, defining the upstream API address, authentication credentials, supported model list, routing weights, and other parameters. A gateway can configure multiple channels (e.g., OpenAI, Azure OpenAI, local vLLM inference services) to achieve unified multi-provider integration. When a user request reaches the gateway, it is distributed to the corresponding channel based on the requested model name and routing rules. See [Channel Fallback](#channel-fallback) and [Provider](#provider). |
| <span id="provider">**Provider**</span> | Provider | A provider is the actual service or system behind a channel that provides LLM inference capabilities, such as OpenAI, Anthropic, Azure, Google, Alibaba Tongyi, Baidu ERNIE, or the platform's internal inference services. Each provider has its specific API protocol and authentication methods, and the gateway adapts protocols through channel configuration. |
| **API Key / Token** | API Key / Token | An API Key (also called Token) is the identity credential for calling the LLM Gateway API. Administrators or developers can create API Keys and set names, expiration times, and callable model scopes for them. Clients include the API Key in request headers to invoke LLM services proxied by the gateway. Each API Key's invocation records are audited. See [Audit Log](#audit-log). |
| <span id="audit-log">**Audit Log**</span> | Audit Log | Audit logs record the details of every API call made through the LLM Gateway, including request time, caller (API Key), requested model, input/output Token counts, response status code, latency, and whether content moderation was triggered. Audit logs help administrators monitor usage, troubleshoot issues, and perform compliance audits. See [Usage Record](#usage-record). |
| **Moderation Policy** | Moderation Policy | Moderation policies define the rules by which the LLM Gateway performs security checks on request and response content. Administrators can create policies at different granularities, specifying check methods (such as keyword matching, regular expressions, AI moderation models), trigger conditions, and actions (block, warn, log, etc.). Moderation policies can be bound at the channel or global level. See [Content Moderation](#content-moderation) and [Lexicon](#lexicon). |
| <span id="lexicon">**Lexicon**</span> | Lexicon | A lexicon is a collection of sensitive words/keywords used in content moderation. Administrators can maintain multiple lexicons, grouping different categories of sensitive words (such as political terms, adult content, advertising, etc.). Lexicons are referenced in moderation policies for keyword matching checks on request or response text. |
| <span id="content-moderation">**Content Moderation**</span> | Content Moderation | Content moderation is the LLM Gateway's security protection capability that checks text content for violations before requests enter upstream models or before responses are returned to users. Moderation can be based on lexicon matching, regex rules, or external moderation AI models. When rules are triggered, requests can be automatically blocked or content can be sanitized. See [Moderation Policy](#moderation-policy). |
| **Rate Limiting (RPM/TPM)** | Rate Limiting (RPM/TPM) | Rate limiting controls the frequency of LLM Gateway API calls. RPM (Requests Per Minute) limits the number of requests per minute, while TPM (Tokens Per Minute) limits the total Token consumption per minute. Rate limiting policies can be configured per API Key, channel, or model dimension to prevent excessive resource consumption or abuse. |
| <span id="channel-fallback">**Channel Fallback**</span> | Channel Fallback | Channel fallback is the LLM Gateway's high availability mechanism. When a channel experiences a failure (e.g., upstream API unavailable, timeout, or high error rate), the gateway automatically routes requests to an alternate channel to ensure uninterrupted service. Administrators can configure fallback order and trigger conditions. See [Channel](#channel). |
| <span id="usage-record">**Usage Record**</span> | Usage Record | Usage records provide aggregated statistics on LLM Gateway resource consumption, including call counts and Token usage per API Key, model, and channel. Usage data can be used for cost accounting, capacity planning, and optimization recommendations. See [Audit Log](#audit-log). |
| **Service Registration** | Service Registration | Service registration is the process of integrating inference services deployed within the platform into the LLM Gateway. Once registered, an inference service appears as a channel in the gateway and can be uniformly invoked, audited, and managed through gateway API Keys without directly exposing the inference service endpoint. See [Inference Service](#inference-service) and [Channel](#channel). |

## Users & Permissions {#users-and-permissions}

The platform uses a role-based multi-level permission system (RBAC), covering three levels: platform, tenant, and workspace.

| Term | English | Description |
|------|---------|-------------|
| <span id="system-admin">**System Admin**</span> | System Admin | The system administrator is the highest-privilege role on the platform, able to perform all operations through the BOSS admin portal, including creating/deleting tenants, registering/managing clusters, configuring global flavors and system templates, managing LLM gateway global configurations, etc. System administrators are typically core members of the platform operations team. See [BOSS (Admin Portal)](#boss). |
| **Tenant Admin** | Tenant Admin | The tenant administrator is responsible for managing all resources and members within their tenant. Permissions include creating/deleting workspaces, inviting/removing members, assigning roles, managing tenant-level LLM gateway configurations, and viewing tenant resource usage. A tenant can have multiple tenant administrators. See [Tenant](#tenant). |
| **Developer** | Developer | A developer is the core user role within a workspace, able to create and manage various AI workload instances (inference services, fine-tuning, development environments, applications, etc.) in authorized workspaces. Developers can read from and write to storage volumes, submit experiments, and perform evaluations, but typically cannot manage workspace members or quota configurations. |
| **Member** | Member | A member is the most basic permission role, with read-only access to workspace resources. Members can view instance statuses, browse logs, view storage volume contents, etc., but cannot create or modify resources. The member role is suitable for personnel who need to understand project progress but do not participate in operations. |
| **RBAC** | Role-Based Access Control | Role-Based Access Control is the core mechanism of the platform's permission management. The platform predefines roles such as system admin, tenant admin, developer, and member, each bound to a set of fine-grained permissions. Permissions are managed separately at the platform, tenant, and workspace levels to ensure the principle of least privilege. See [Permission Expression](#permission-expression). |
| <span id="permission-expression">**Permission Expression**</span> | Permission Expression | Permission expressions (e.g., `rune:inference:create`, `moha:model:read`) are the encoding format for fine-grained permissions in the platform. Each expression consists of three parts: "sub-product:resource-type:action", used to precisely control a user's operational capabilities on specific resources. Roles are composed of a set of permission expressions. |
| **MFA** | Multi-Factor Authentication | Multi-Factor Authentication (MFA) is a mechanism that enhances login security by adding a second verification factor (such as a TOTP dynamic verification code) beyond username/password. The platform supports enabling MFA for user accounts, requiring both a password and a dynamic verification code during login to effectively prevent account compromise. |
| **API Key (User-Level)** | API Key (User-Level) | User-level API Keys are identity credentials for programmatic access to the platform's REST API, distinct from LLM Gateway API Keys. Users can generate API Keys in their personal settings, set validity periods and permission scopes, then use them in CLI tools or automation scripts. See [API Key / Token](#api-key--token). |
| **SSH Key** | SSH Key | SSH Keys are a public-key cryptography-based authentication method. After uploading their public key to the platform, users can perform Git repository push/pull operations via the SSH protocol (without entering a password each time). SSH Keys are particularly common in Git operations with Moha model repositories and dataset repositories. |

## Compute Resources {#compute-resources}

The platform's compute resource management is organized around four core concepts: clusters, resource pools, flavors, and quotas.

| Term | English | Description |
|------|---------|-------------|
| <span id="flavor">**Flavor**</span> | Flavor | A flavor is a predefined compute resource configuration template that defines the combination of CPU cores, memory size, and accelerator (GPU/NPU, etc.) count available to an instance. For example, "4C 16G 1×A100" means 4 CPU cores, 16 GB memory, and 1 A100 GPU. Platform administrators manage flavor templates centrally in BOSS, and tenants and developers select from available flavor lists when creating instances. See [Quota](#quota). |
| <span id="resource-pool">**Resource Pool**</span> | Resource Pool | A resource pool is a logical grouping of nodes within a cluster, used for resource isolation and differentiated scheduling. For example, GPU nodes and CPU nodes can be placed in different resource pools, or dedicated resource pools can be created for different tenants. Workspaces are bound to a specific resource pool at creation, and all instances within them can only be scheduled to nodes in that resource pool. See [Node](#node). |
| <span id="quota">**Quota**</span> | Quota | Quotas are upper-limit controls on resource usage, preventing a single tenant or workspace from over-consuming shared resources. The platform supports a three-level quota system: **cluster-level quota** (limiting a tenant's total resources in a cluster), **tenant-level quota** (limiting a tenant's global resources), and **workspace-level quota** (limiting a specific workspace's resources). Quotas are typically measured in CPU cores, memory GB, and accelerator count. |
| **GPU** | Graphics Processing Unit | GPUs (Graphics Processing Units) are the most commonly used acceleration hardware for AI training and inference. The platform supports various NVIDIA GPU models (such as A100, H100, V100), managed and allocated through resource pools and flavors. |
| **NPU** | Neural Processing Unit | NPUs (Neural Processing Units) are specialized chips optimized for AI inference and training, such as the Huawei Ascend series. The platform supports NPUs in a manner similar to GPUs, managed through flavors and resource pools. |
| **DCU** | Deep Computing Unit | DCUs (Deep Computing Units) are general-purpose GPU accelerator cards from Hygon Information, used as alternatives to NVIDIA GPUs in domestic hardware scenarios. The platform supports scheduling and management of DCU accelerator cards. |
| **MLU** | Machine Learning Unit | MLUs (Machine Learning Units) are AI acceleration chips from Cambricon. The platform manages MLUs as an accelerator type alongside GPUs for resource management and scheduling. |
| **vGPU** | Virtual GPU | vGPU (Virtual GPU) is a technology that virtualizes a single physical GPU into multiple logical GPUs, allowing multiple instances to share the computing power and memory of one physical GPU. vGPU is suitable for scenarios with smaller GPU resource requirements (such as lightweight inference, development debugging) and can significantly improve GPU utilization. |
| **Pod** | Pod | A Pod is the smallest schedulable unit in Kubernetes, containing one or more containers. Each instance (inference service, fine-tuning task, etc.) in the platform runs as a Pod in the K8s cluster. Users typically do not need to operate Pods directly but can view Pod status and logs in instance details. |
| <span id="node">**Node**</span> | Node | A node is a physical or virtual server in a Kubernetes cluster. Nodes provide compute resources such as CPU, memory, and accelerators, and Pods are scheduled to run on nodes. Nodes can be classified by role as master nodes and worker nodes. See [Cluster](#cluster). |

## Application Templates & Market {#application-templates-and-market}

The platform uses a template-based mechanism to achieve standardized application deployment and distribution.

| Term | English | Description |
|------|---------|-------------|
| <span id="product-template">**Product Template**</span> | Product (Template) | A product template is a blueprint for application deployment, defining the container image, resource requirements, configuration parameters, and deployment method. Each template is built on a Helm Chart and includes version management. Templates are divided into system templates (visible to all tenants) and admin templates (visible to specific tenants). See [Helm Chart](#helm-chart). |
| <span id="helm-chart">**Helm Chart**</span> | Helm Chart | A Helm Chart is the Kubernetes application packaging standard that encapsulates YAML resource manifests, configuration templates, and default parameters into a reusable deployment package. XiaoShi AI's product templates use the Helm Chart format underneath, supporting version releases and upgrade rollbacks. |
| **SchemaForm (Dynamic Form)** | SchemaForm | SchemaForm is a dynamic form automatically generated by the platform based on a product template's configuration parameter definitions (JSON Schema). When deploying applications, users don't need to manually edit YAML files. Instead, they fill in configuration parameters (such as replica count, image version, environment variables) through a visual form, lowering the deployment barrier. |
| **Version** | Version | A product template's version number (e.g., `1.0.0`, `2.3.1`) identifies a specific release. Each time template content is updated, a new version is released. Users can select a specific version during deployment or upgrade to the latest version. Version management ensures traceability and stability of template changes. |
| <span id="app-market">**App Market**</span> | App Market | The App Market is the template browsing and deployment entry point for developers and workspace administrators. Users can search and filter available product templates, view template descriptions and configuration parameters, then deploy to a specified workspace with one click. The App Market enables self-service application distribution. See [Product Template](#product-template). |
| **System Template** | System Product | System templates are product templates created and managed by platform administrators through BOSS, visible to all tenants. System templates typically include platform-recommended, verified standard applications (such as common inference frameworks, monitoring tools, ML pipeline components, etc.). |
| **Admin Template** | Admin Product | Admin templates are product templates created and managed by tenant administrators, visible only to workspaces under that tenant. Admin templates allow tenants to customize proprietary application templates based on their business needs, such as internal tools and private services. |

## Storage & Files {#storage-and-files}

The platform provides multi-layered storage management capabilities to support data persistence and file management needs for AI workloads.

| Term | English | Description |
|------|---------|-------------|
| **Storage Volume** | Storage Volume | See [AI Workloads - Storage Volume](#storage-volume). Storage volumes are persistent storage media for instances that can share data across multiple instances. |
| <span id="storage-cluster">**Storage Cluster**</span> | Storage Cluster | A storage cluster is the platform's underlying distributed storage infrastructure that provides actual storage space for storage volumes. Platform administrators register and manage storage clusters in BOSS. A compute cluster can be associated with multiple storage clusters to meet different capacity and performance requirements. |
| **S3 File Proxy** | S3 File Proxy | The S3 File Proxy is the platform's file access proxy service that exposes underlying storage (such as Ceph, MinIO, etc.) files through an S3-compatible interface to users or instances. Users can upload, download, and manage files in storage volumes through the platform's file manager or S3 client tools. |
| **File Manager** | File Manager | The File Manager is the platform's built-in web file browsing tool. Users can browse storage volume directory structures, preview file contents, upload/download files, and create/delete folders through a visual interface. The File Manager eliminates the need for users to manage files through command-line operations. |
| **Storage Job Types** | Storage Job Kinds | The platform supports the following storage job types:<br/>• **Git**: Clone code or files from a Git repository to a storage volume.<br/>• **HuggingFace**: Download models or datasets from Hugging Face Hub to a storage volume.<br/>• **ModelScope**: Download models or datasets from ModelScope to a storage volume.<br/>• **PythonEnv**: Create a Python virtual environment in a storage volume and install specified dependencies.<br/>• **Moha**: Sync models or datasets from the platform's built-in Moha repository to a storage volume.<br/>See [Storage Job](#storage-job). |

## Observability {#observability}

The platform provides comprehensive observability capabilities to help users and administrators monitor resource usage and troubleshoot operational issues.

| Term | English | Description |
|------|---------|-------------|
| **Dashboard** | Dashboard | The dashboard is the platform's overview information page, displaying key statistics and resource summaries. The Console dashboard shows tenant-level workspace overview, instance statistics, and resource usage; the BOSS dashboard shows platform-level cluster status, tenant counts, and global resource usage. |
| **Dynamic Dashboard** | Dynamic Dashboard | The dynamic dashboard is a customizable monitoring panel in BOSS based on Grafana or a similar engine. Administrators can configure custom metric queries and charts to display real-time cluster CPU/GPU utilization, memory consumption, Pod scheduling status, and other key operational metrics. See [Metrics](#metrics). |
| **Log Viewer** | Log Viewer | The Log Viewer is the log browsing tool on the instance detail page. Users can view stdout/stderr logs of containers within instances, with support for keyword search, time range filtering, and log download. The Log Viewer helps developers quickly troubleshoot errors and anomalies during instance runtime. See [Log Stream](#log-stream). |
| <span id="log-stream">**Log Stream**</span> | Log Stream (WebSocket) | Log streaming uses the WebSocket protocol for real-time push of instance logs. When users open the Log Viewer, new log entries are appended to the page in real time without manual refresh. This is particularly useful for monitoring running training tasks or service startup processes. |
| <span id="metrics">**Metrics**</span> | Metrics | Metrics are time-series data that quantify system operational states, such as CPU utilization, GPU memory usage ratio, network throughput, and request latency. The platform collects metric data from Kubernetes and various instances, viewable in dynamic dashboards and instance details. Metric data is typically collected and stored using the Prometheus ecosystem. |
| **Events** | Events | Events are system event information recorded in Kubernetes, such as Pod scheduling success/failure, image pull completion, and container restarts. Users can view related events on the instance detail page to understand the creation process and runtime state changes. Event information is very helpful for troubleshooting instance startup failures. |

The platform provides monitoring support at multiple levels: cluster-level monitoring (through dynamic dashboards), instance-level monitoring (through Log Viewer and metrics charts), and gateway-level monitoring (through audit logs and usage records). |
